# Анализ закупок группы Сбер за 2024–2025 годы

Проект собирает, связывает, очищает и анализирует открытые данные о закупках
юридических лиц группы Сбер. Основной результат — воспроизводимый аналитический
отчёт в Jupyter Notebook, дополненный PostgreSQL, SQL-запросами, графиками и
локальным LLM-анализом.

## Основной отчёт

[Открыть Jupyter Notebook](notebooks/final_sber_procurement_report.ipynb)

Дополнительная переносимая версия:

[Открыть HTML-отчёт](data/processed/stage4/stage4_report.html)

В репозитории оставлен один Notebook. Все восемь графиков встроены непосредственно
в файл как сохранённые результаты ячеек, поэтому видны сразу после открытия.

## Что необходимо установить

### Python 3.12

Скачать:

`https://www.python.org/downloads/windows/`

При установке включить `Add Python to PATH`. Затем запустить:

```powershell
.\setup_environment.cmd
```

Будут установлены pandas, numpy, matplotlib, seaborn, scipy, Jupyter Lab,
psycopg и остальные зависимости из `requirements.txt`.

### PostgreSQL 16

Скачать:

`https://www.postgresql.org/download/windows/`

Во время установки сохранить пароль пользователя PostgreSQL. В проекте по
умолчанию используются:

```text
сервер: localhost
порт: 5432
база: Sber
пользователь: admin
```

Если база ещё не создана, выполнить `stages/stage2/sql/create_database.sql`
через pgAdmin или `psql`. После этого запустить:

```powershell
.\update_database.cmd
```

### Ollama и Qwen

Ollama нужна только для бесплатного локального LLM-анализа.

Скачать:

`https://ollama.com/download/windows`

После установки открыть PowerShell:

```powershell
ollama --version
ollama pull qwen3.5:9b
```

Для видеокарты с 8 ГБ VRAM используется `qwen3.5:9b`. Модель занимает примерно
6,6 ГБ. Запросы выполняются локально и не требуют API-ключа.

Проверка:

```powershell
ollama list
```

В списке должна присутствовать `qwen3.5:9b`.

Запуск следующих 25 задач:

```powershell
.\run_stage3_qwen.cmd
```

Или:

```powershell
.\run_stage3_qwen.ps1 -Limit 25
```

Результаты сохраняются после каждой задачи в
`data/processed/stage3/llm_results.jsonl`. Повторный запуск продолжает очередь
и не обрабатывает уже сохранённые задачи.

## PostgreSQL

Схема называется `sber_procurement`.

| Таблица | Назначение | Строк |
|---|---|---:|
| `organizations` | Реестр юридических лиц группы | 17 |
| `purchases` | Одна строка на каноническую закупку | 1 458 |
| `purchase_sources` | Карточки закупки на разных площадках | 1 461 |
| `purchase_candidates` | Неподтверждённые поисковые результаты | 6 302 |
| `duplicate_audit` | Аудит межисточниковых дублей | 3 |
| `documents` | Очередь карточек и приложенных документов | 1 461 |

Связи построены через `canonical_purchase_id`. Таблицы источников, дублей и
документов ссылаются на каноническую закупку. Представления
`v_monthly_dynamics` и `v_customer_summary` используются для аналитики.

Обновление базы:

```powershell
.\update_database.cmd
```

SQL-файлы:

- `stages/stage2/sql/schema.sql` — структура;
- `stages/stage2/sql/load_data.psql` — загрузка CSV;
- `stages/stage2/sql/analytics_queries.sql` — аналитические запросы;
- `stages/stage2/sql/demo_queries.sql` — демонстрация базы.

## Аналитические запросы

Реализованы запросы для:

- месячной динамики;
- концентрации закупок по заказчикам;
- структуры способов закупки;

- аудита дублей;

- очереди ручной проверки;
- очереди обработки документов;
- поиска текстовых признаков единственного поставщика;
- поиска похожих повторных процедур.

## Аналитический модуль

Ключевое направление — IT и телеком:

- 210 процедур;
- 906,82 млн руб. указанной стоимости;
- рост количества с 61 до 149 процедур;
- рост суммы с 199,35 до 707,47 млн руб.

Проверены гипотезы о связи:

- IT-закупок с USD/RUB;
- строительных закупок с ключевой ставкой.

Для сумм используется `log1p`, коэффициент Пирсона проверяется перестановочным
тестом на 10 000 перестановок.

## Аномалии и LLM

Выявляются:

- экстремальные суммы;
- отменённые процедуры;
- отмена с последующей похожей публикацией;
- двукратные и более изменения цены похожего предмета;
- конфликт статуса и текста карточки.

Локальная Qwen 3.5 9B используется через Ollama. Она формирует JSON и черновики
выводов, но не считается источником фактов о нарушениях, победителях или
единственном поставщике.

Запуск следующей порции LLM-задач:

```powershell
.\run_stage3_qwen.ps1 -Limit 25
```

## Визуализации

Финальный Notebook содержит:

1. динамику закупок по месяцам;
2. структуру направлений;
3. топ-20 крупнейших процедур;
4. IT-закупки и USD/RUB;
5. строительство и ключевую ставку;
6. распределение указанных сумм;
7. доступность НМЦ и итоговой цены;
8. аудит качества Qwen.

У каждого блока есть наблюдение, интерпретация, значимость и ограничение.

## Ограничения

- итоговая цена контракта отсутствует;
- участники и победители не извлечены из протоколов;
- часть сумм является тарифом или единичной расценкой;
- классификация направлений основана на словаре;
- 2024 и 2025 годы имеют различное покрытие источника;
- корреляция не доказывает причинную связь.

## Как представить проект в Jupyter Notebook

1. Открыть `notebooks/final_sber_procurement_report.ipynb`.
2. Убедиться, что восемь встроенных графиков отображаются сразу.
3. При необходимости выполнить `Trust Notebook`.
4. Выполнить `Restart Kernel and Run All Cells`.
5. Скрыть технические кодовые ячейки после выполнения.
6. Включить режим презентации или полноэкранный режим браузера.
7. Последовательно показать:
   - постановку задачи и источники;
   - процесс сбора и обезличивания;
   - структуру PostgreSQL;
   - примеры SQL;
   - гипотезы;
   - восемь визуализаций;
   - выводы и ограничения.
8. На каждом графике проговорить четыре пункта: наблюдение, интерпретация,
   значимость и ограничение.
9. Отдельно подчеркнуть, что итоговая цена контракта не была выдумана, а её
   отсутствие зафиксировано как ограничение данных.
10. Завершить демонстрацию ролью локальной Qwen и объяснить, что LLM формирует
   черновик, но не подтверждает нарушения.
