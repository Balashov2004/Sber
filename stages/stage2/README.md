# Этап 2. Обработка и PostgreSQL

## Входные данные

- канонические закупки после связывания источников;
- точные связи ЕИС ↔ Сбербанк-АСТ;
- обезличенные строгие и поисковые выгрузки;
- реестр юридических лиц группы Сбер.

## Очистка и дубли

Единица аналитики — одна закупочная процедура. Дубли удаляются только при
наличии точного номера закупки и подтверждённой межисточниковой связи.
Совпадение названия, даты или суммы не является основанием для удаления.

Результат:

- 1 458 канонических закупок;
- 1 461 строка источников;
- 3 межисточниковых дубля;
- доля дублей 0,2053%;
- 6 302 кандидата для ручной проверки;
- 17 организаций.

Запуск:

```powershell
python stages\stage2\scripts\process_stage2.py
```

## Структура базы

Схема PostgreSQL: `sber_procurement`.

### `organizations`

Юридические лица группы Сбер: ИНН, КПП, ОГРН, наименование, связь с группой и
источник проверки.

### `purchases`

Каноническая таблица закупок. Одна строка соответствует одной процедуре.
Содержит номер, предмет, заказчика, даты, указанную сумму, статус, способ
закупки, источник и ссылки.

### `purchase_sources`

Представления одной закупки на разных площадках. Связана с `purchases` через
`canonical_purchase_id`.

### `purchase_candidates`

Поисковые результаты, где принадлежность заказчика группе Сбер ещё не
подтверждена. Эти строки не смешиваются с основной аналитикой.

### `duplicate_audit`

Аудит объединённых межисточниковых записей: метод связи, уверенность,
сохранённый и удалённый источник.

### `documents`

Очередь карточек и приложенных документов. Хранит статусы скачивания,
извлечения текста, обезличивания и LLM-обработки.

## Связи

```text
organizations

purchases
  ├── purchase_sources
  ├── duplicate_audit
  └── documents

purchase_candidates
```

## Представления и индексы

- `v_monthly_dynamics` — число процедур, сумма и медиана по месяцам;
- `v_customer_summary` — число и сумма закупок по заказчикам;
- индексы по дате, ИНН заказчика, сумме и году/месяцу.

## SQL-запросы

`sql/analytics_queries.sql` содержит запросы для:

1. контроля покрытия и заполненности;
2. сравнения 2024 и 2025 годов;
3. месячной динамики;
4. концентрации по заказчикам;
5. структуры способов закупки;
6. топ-20 крупнейших процедур;
7. статистики дублей;
8. сверки карточек между источниками;
9. очереди ручной проверки;
10. очереди документов;
11. текстовых признаков единственного поставщика;
12. поиска похожих повторных процедур.

## Документы

Для приложений предусмотрен процесс:

```text
ссылка → скачивание → SHA-256 → извлечение текста/OCR
→ обезличивание → структурирование → LLM-анализ
```

Из протоколов необходимо дополнительно получить участников, победителя,
итоговую цену, дату договора и критерии оценки.
